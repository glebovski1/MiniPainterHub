## Reflection on Google authentication delivery

### What went well

- Kept JWT bearer as the API default while using an explicit Google challenge and the same token issuer for password and external accounts.
- Proved the sensitive lifecycle end to end with a deterministic fake provider: onboarding, return login, conflict, explicit link, password setup, disconnect, cancellation, expiry, registration controls, suspension, and return-path safety.
- Repeated the full visual sweep after inspecting the generated captures instead of treating screenshot generation alone as review.

### Possible bad patterns detected

- In-memory tests initially missed SQL Server's retry-strategy restriction on user-created transactions.
- Browser URI behavior exposed a runtime-specific root-relative parsing difference that unit tests did not reproduce.
- The first visual pass showed a missing external Google logo and an invalid space-containing suggested username.

### Fixes applied now

- Wrapped relational registration and linking in the configured EF execution strategy while preserving atomic transactions.
- Made return-path parsing browser-runtime safe and added real-browser coverage for password and Google navigation.
- Stored the official Google mark locally, sanitized username suggestions against Identity's configured character set, and repeated all 74 captures until clean.

### Fixes recommended later

- Owner-review Privacy and Terms before activating Google with real credentials.
- Complete the documented Google Testing-audience and Azure configuration smoke before enabling the feature flag.
